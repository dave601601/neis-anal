## 작업 흐름: "고" 신호 전까지 구현 보류

내가 명시적으로 "고"(또는 동등한 승인)를 말하기 전까지는 구현을 시작하지 않는다. 기본 행동은 구체적인 계획을 제시한 뒤 기다리는 것이다.

- "구현"이란 코드/설정 작성·수정, 파일 생성, 빌드·커밋 실행을 뜻한다.
- 승인 없이도 항상 가능한 일: 코드 읽기, 검색, 질문 답변, 계획 제안.
- 작업 도중에도 동일하게 적용된다 — 승인된 한 단계를 마쳤다고 해서 다음 단계 착수가 자동 승인되는 것은 아니다.

## 언어 정책 (프로젝트 전용 예외)

이 프로젝트는 도메인과 원천 데이터가 모두 한국어다: `opendata.mnd.go.kr`의 군 급식 데이터, 지역 특산물, 부대 명칭, 급양대 / 간부 / 식단 / 칼로리 등의 용어. 영어 번역은 손실이 크고 데이터 매칭을 놓치게 한다.

이 프로젝트에 한해서 일반적인 "영어 우선" 기본값을 다음과 같이 뒤집는다.

- **사고는 한국어로.** 내부 추론, 가설 정리, 단계 계획은 데이터에 등장하는 한국어 용어를 그대로 쓴다 (예: 급양대, 특산물, 간부, 식단). 번역해서 사고하지 않는다.
- **검색도 한국어로.** 웹 검색, 공공데이터포털 질의, 한글 텍스트에 대한 `grep`, 컬럼명 조회 모두 한글 키워드를 우선한다. 결과가 비었을 때만 영어로 대체한다.
- **산출물도 한국어로.** 문서, 코드 주석, 커밋 메시지 본문, PR 설명 — 영구 산출물도 한국어로 작성한다. 단 아래 "글쓰기 스타일" 절의 제한된 영어 유지 항목(예: Conventional Commits 접두어)은 그대로 둔다.
- **대화 회신**은 사용자가 사용한 언어를 따른다.

## 글쓰기 스타일 (영구 산출물)

적용 범위: 문서 (`README.md`, `docs/**`, 코드 주석), 커밋 메시지, PR 설명, 런치 파일 주석 등 저장소에 들어가는 모든 산출물. 대화 회신에는 적용하지 않는다.

- **한국어로 쓴다.** 위 "언어 정책"에 따라 본문은 한국어가 기본이다. 단 다음 항목은 영어/원어를 유지한다.
  - Conventional Commits 접두어 (`feat:`, `fix:`, `refactor:`, `perf:`, `docs:`, `test:`, `chore:`)
  - 코드 식별자, 라이브러리 이름, API 명칭, 파일 경로
  - 영어가 정착된 기술 용어 (예: PPO, observation masking) — 한국어로 옮겼을 때 모호해지면 원어를 쓴다
- **꾸밈 없이 본론으로.** 산문에서 `**굵게**`를 쓰지 않는다. 표 셀에서 의미상 강조가 필요할 때(예: 지표 컬럼의 우승값)만, 그것도 아껴서 쓴다.
- **직접 진술한다.** "참고로 …", "유의할 점은 …" 같은 도입을 빼고 사실부터 적는다. 군더더기를 줄인다.

## Git 커밋 정책

작업 진행에 맞춰 코드 변경을 자동으로 커밋한다. 아래 규칙을 엄격히 따른다.

### 커밋 시점
- 논리적 단위(기능, 수정, 리팩터)가 완료된 직후 — 한 줄 한 줄마다는 아니다.
- 변경 영역의 테스트가 통과한 직후.
- 무관한 작업으로 맥락을 전환하기 직전.
- 작업 트리가 더러운 채로 세션을 끝내지 않는다.

### 커밋 메시지 형식
Conventional Commits를 사용한다.
- `feat: <추가된 기능>`
- `fix: <수정된 버그>`
- `refactor: <재구성한 부분>`
- `perf: <성능 변경>`
- `docs: <문서 변경>`
- `test: <테스트 변경>`
- `chore: <도구·의존성·설정>`

규칙:
- 제목은 72자 이내, 동사형으로 시작, 끝에 마침표를 붙이지 않는다.
- 비자명한 변경은 빈 줄을 두고 본문에 *왜* 했는지를 적는다. *무엇을* 했는지가 아니다.
- 유용할 때는 스코프 접두어를 쓴다: `feat(ppo): 관측 마스킹 추가`.

### 스테이징
- 명시적 경로 또는 `git add -p`를 우선한다. `git status`를 먼저 확인하지 않고 `git add .` 또는 `git add -A`를 쓰지 않는다.
- 커밋은 원자적으로 — 논리 변경 하나당 커밋 하나. 무관한 수정은 분리한다.

### 커밋 전 점검
1. `git status` + `git diff --staged`로 범위를 확인한다.
2. 프로젝트 테스트 명령이 있다면 실행한다.
3. 린터·포매터가 설정되어 있다면 실행한다.
4. 위 항목 중 실패가 있으면 커밋을 중단하고 먼저 고친 뒤 커밋한다.

### 절대 커밋하지 않는 것
- 비밀값, API 키, 토큰, `.env*` 파일.
- 큰 바이너리, 데이터셋, 체크포인트, rosbag, 모델 가중치 (>10 MB). 대신 `.gitignore`에 추가한다.
- 디버그용 출력문, `TODO: remove`, 실수로 남은 주석 처리된 코드.
- 깨진 코드를 `main`에 올리지 않는다. 중간 체크포인트가 필요하면 WIP 브랜치를 쓴다.

### 명시적 확인 없이는 절대 하지 않는 것
- 공유 브랜치에 대한 `git push --force` / `--force-with-lease`.
- `git reset --hard`, `git clean -fdx` 등 커밋되지 않은 작업을 버리는 명령.
- 이미 푸시된 커밋에 대한 히스토리 재작성 (`rebase -i`, `commit --amend`).
- 브랜치 삭제·이름 변경.

### 기여자 표기
커밋 메시지에 `Co-authored-by: Claude` 등 AI 기여자 트레일러를 추가하지 않는다.

### 푸시 정책
자동으로 푸시하지 않는다. 로컬에 커밋만 하고, 준비되면 내가 직접 푸시한다.

## 진행 상황 추적

`docs/` 아래에 상시 진행 로그를 유지한다. 이 로그는 "현재 어디까지 왔는가"에 대한 단일 출처다 — git 히스토리(*무엇이* 바뀌었는지)나 CLAUDE.md(*변하지 않는 규칙*)와는 별개다.

구조:
- `docs/PROGRESS.md` — 인덱스만 (포인터 + 현재 `## Open` 목록 + 작성 규칙). 50줄 이내로 유지.
- `docs/progress/<topic>.md` — 주제별 작업 로그. 실제 항목은 여기에 들어간다.
- `docs/SIM_TO_REAL.md`(및 유사 계획 문서) — `[ ]` 체크박스를 포함한 미래 계획서. 작업 로그가 아니다.

### 세션 시작 시
세션 첫머리에 매번:
1. `docs/PROGRESS.md`를 읽어 주제 목록과 `## Open`을 확인한다. 현재 작업과 관련된 주제 파일을 연다.
2. `PROGRESS.md`가 아직 없다면 기존 양식을 참고해 인덱스로 만든다.
3. 다음 단계를 제안하기 전에 현재 상태를 짧게 요약해 보여준다.

### 진행 로그를 갱신할 시점
- **로그가 설명하는 코드 변경과 같은 커밋에 함께 스테이징한다.** 코드와 진행 로그 항목을 같은 커밋에 묶는다 — 논리 단위 하나당 커밋 하나이며, 별도의 `docs(progress)` 커밋을 따로 만들지 않는다. 항목은 *무엇*과 *왜*를 기록하면 충분하고, SHA를 인용할 필요는 없다(커밋 자체가 링크다).
- 실험이 끝났을 때 — 성공·실패·미결 모두 기록한다.
- 설계 결정을 내렸을 때. *무엇*만이 아니라 *왜*를 기록한다.
- 자명하지 않은 사실을 발견했을 때 (버그 원인, 라이브러리 특이사항, 기준 수치 등).
- 블로커가 생기거나 해소됐을 때.

### 새 항목 위치
- 새 항목은 해당 `docs/progress/<topic>.md`의 **맨 위**에 추가한다(파일 내에서는 최신이 먼저).
- 맞는 주제 파일이 없으면 새로 만들고 `docs/PROGRESS.md`에서 링크한다.
- 주제 파일이 500줄을 넘어가면 분리를 고려한다.

주제 파일 안에서 다음 구조를 사용한다.
